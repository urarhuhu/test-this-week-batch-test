Docusaurus 在九個多月前[正式發布](https://v1.docusaurus.io/blog/2017/12/14/introducing-docusaurus)，作為一種輕鬆構建開源文檔網站的方式。自那時起，它已累積超過 8,600 個 GitHub Stars，並被許多流行的開源項目使用，例如 [React Native](https://facebook.github.io/react-native/)、[Babel](https://babeljs.io/)、[Jest](https://jestjs.io/)、[Reason](https://reasonml.github.io/) 和 [Prettier](https://prettier.io/)。

有句話說，最好的軟體是不斷演進的，而最差的則停滯不前。如果你還不知道，我們一直在計劃並著手開發 Docusaurus 的下一個版本 🎉。

## 基礎架構

### 內容

Docusaurus 1 的網站實際上是由一堆靜態 HTML 頁面構建而成。儘管使用了 React，我們並未充分利用 React 提供的功能，例如組件狀態，這允許動態和互動式頁面。React 僅被用作靜態內容的模板引擎，而互動性必須通過腳本標籤和 `dangerouslySetInnerHTML` 來添加 😱。

此外，改變 Docusaurus 加載內容的方式並不容易。例如，原生不支持添加 CSS 預處理器如 Sass 和 Less，這需要用戶通過添加自定義腳本進行許多變通。

在 Docusaurus 2 中，我們將使用 [webpack](https://webpack.js.org/) 作為模塊打包工具，並改變內容的提供方式。添加 CSS 預處理器將像添加一個 webpack 加載器一樣簡單。與純靜態 HTML 不同，**在構建時我們將創建一個服務器渲染版本的應用**，並渲染相應的 HTML。Docusaurus 網站本質上將是一個同構/通用應用程序。這種方法深受 [Gatsby](https://github.com/gatsbyjs/gatsby) 的啟發。

### 版本控制

如果你使用 Docusaurus 有一段時間，你可能會注意到 Docusaurus 創建版本化文檔**僅當且僅當**文檔內容**不同**時。

例如，如果我們有 `docs/hello.md`：

```text
---
id: hello
title: hello
---
Hello world !
```

而**我們創建版本 1.0.0**，Docusaurus 將創建 `versioned_docs/version-1.0.0/hello.md`：

```text
---
id: version-1.0.0-hello
title: hello
original_id: hello
---
Hello world !
```

然而，如果在創建 v2.0.0 時 `hello.md` 沒有變化，Docusaurus 將不會為該文檔創建任何版本化文檔。換句話說，`versioned_docs/version-2.0.0/hello.md` 將不存在。

這對用戶來說可能非常困惑；如果他們想編輯 v2.0.0 的文檔，他們必須編輯 `versioned_docs/version-1.0.0/hello.md` 或手動添加 `versioned_docs/version-2.0.0/hello.md`。這可能導致意外的錯誤。這裡有一個 [Jest 中的真實案例](https://github.com/facebook/jest/pull/6758#issuecomment-408274413)。

此外，這增加了程式碼庫的複雜性，因為我們需要一個版本回退機制。在構建時，Docusaurus 必須替換為正確版本的連結。這也導致了一個錯誤：[重新命名文件會破壞舊版本中的連結](https://github.com/facebook/docusaurus/issues/845)。

在 Docusaurus 2 中，**每次我們發布新版本時，我們將改為對所有文件進行快照**。我們不會要求文件的內容必須有所變更。這是為了更好的開發者和使用者體驗而做出的空間複雜度權衡。我們將使用更多空間來實現更好的關注點分離和保證正確性。

### 翻譯

Docusaurus 通過使用 [Crowdin](https://crowdin.com/) 提供了簡單的翻譯功能。用英文撰寫的文件會被上傳到 Crowdin，由社群中的使用者進行翻譯。我們一直假設**英文**是預設語言，但這可能並非所有使用者的情況。我們已經看到許多非英文的開源專案使用 Docusaurus。

在 Docusaurus 2 中，**我們不會假設英文是預設語言**。當使用者啟用國際化功能時，他們必須在 `siteConfig.js` 中設定一個預設語言。我們將假設 `docs` 目錄中的所有文件都是用該語言撰寫的。

此外，在完成 Docusaurus 2 的最小可行產品（MVP）後，我意識到可以不用 Crowdin 進行翻譯。因此，我們可能需要新增一個工作流程來支援這種情況。不過，我們仍然強烈建議使用 Crowdin 以實現更簡單的整合。

## 可自訂性

### 佈局

目前 Docusaurus 的狀態是它負責整個佈局和樣式，這無意中讓使用者很難根據自己的意願自訂網站的外觀。

在 Docusaurus 2 中，**佈局和樣式應由使用者控制**。Docusaurus 將處理內容生成、路由、翻譯和版本控制。受 [create-react-app](https://github.com/facebook/create-react-app) 和 [VuePress](https://vuepress.vuejs.org/) 的啟發，Docusaurus 仍將提供一個預設主題，使用者可以彈出（eject）以進一步自訂佈局和樣式。這意味著使用者甚至可以通過使用 [React Helmet](https://github.com/nfl/react-helmet) 來更改 HTML 元數據。社群驅動的主題也非常有可能。這種讓使用者控制佈局和樣式的方法被大多數靜態網站生成器所採用。

### Markdown

我們目前的 Markdown 解析由 [Remarkable](https://github.com/jonschlinkert/remarkable) 提供支援。如果使用者想使用 [Markdown-it](https://github.com/Markdown-it/Markdown-it) 或甚至 [MDX](https://github.com/mdx-js/mdx) 呢？然後還有語法高亮器的選擇問題（例如：[Prism](https://prismjs.com/) vs [Highlight.js](https://highlightjs.org/)）。我們應該將這些選擇權留給使用者。

在 Docusaurus 2 中，**使用者可以彈出並選擇自己的 Markdown 解析器**。無論他們想使用另一個 Markdown 解析器如 [Remark](https://github.com/remarkjs/remark)，還是甚至自己的內部 Markdown 解析器，都沒有關係。作為一個經驗法則，使用者必須提供一個 React 組件，我們將在其中提供一個包含 _Markdown 原始字串_ 的 children 屬性。預設情況下，我們將使用 Remarkable 作為 Markdown 解析器，並使用 Highlight.js 進行語法高亮。預設解析器將來仍可能變更，因為我們仍在試驗不同的 Markdown 解析器。

### 搜尋

我們的核心搜尋功能基於 [Algolia](https://www.algolia.com/)。有使用者要求能夠使用不同的搜尋服務，例如用於離線搜尋的 `lunrjs`。

我個人非常喜歡 Algolia，與他們合作的體驗相當良好。他們的回應速度很快，而且由於他們的 `DocSearch` 是開源的，我們可以輕鬆提交 Pull Request。例如，我最近提交了[這個 PR，讓 DocSearch 能夠抓取 sitemap 中的替代語言](https://github.com/algolia/docsearch-scraper/pull/387)。

在 Docusaurus 2 中，**我們將允許用戶自訂搜尋框**。用戶只需從預設主題中彈出並修改搜尋 UI（一個 React 元件）即可。不過，預設主題仍會使用 Algolia。

## 穩定性

軟體永遠不會完美，但我們希望 Docusaurus 在新增功能時不會崩潰。Docusaurus 最初發布時，並沒有強大的自動化測試套件，因此許多回歸問題未能及早發現。雖然我們最近添加了許多測試，但測試覆蓋率仍然相對較低。

在 Docusaurus 2 中，**我們會在開發過程中逐步添加測試**，因為我們正在進行全面重寫。因此，我相信它會比以往更加穩定，相較於 Docusaurus 1，破壞功能的可能性也會更低。

## 常見問題

### 是否會有破壞性變更？

如果你已經讀到這裡，應該能注意到將會有一些破壞性變更。雖然我們會盡量**減少破壞性變更的數量**並盡可能保持向後兼容，但我們認為某些破壞性變更是必要的。這主要是因為 Docusaurus 2 是對程式碼庫的**重大重寫與重新架構**。

由於開發尚未 100% 完成，確切的破壞性變更清單目前還不完全清楚。不過，我想強調的一點是，我們將棄用 `siteConfig.js` 中的許多選項，並計劃盡量保持其精簡。例如，`cleanUrl` 這個 siteConfig 將被棄用，因為 Docusaurus 2 的所有網址都不會帶有 `.html` 後綴。

我們的目標是讓大多數網站能夠無痛升級到 Docusaurus 2。發布 Docusaurus 2 時，我們也會提供遷移指南。屆時，歡迎在 [Discord](https://discord.gg/docusaurus) 或 [X](https://x.com/docusaurus) 上聯繫我們提問或尋求協助。

### Docusaurus 2 何時發布？

目前，我們還沒有確切的發布日期。我個人估計可能會在未來一到兩個月內發布 Alpha 版本，但這當然只是預估。

我想分享的是，雖然 Docusaurus 是 [Facebook 開源](https://opensource.fb.com/)的一部分，且團隊成員多數是 Facebook 員工，但維護與開發工作大多是在正常工作時間之外進行的。我目前是 [新加坡南洋理工大學](https://x.com/NTUsg) 的大四學生，因此必須在課程作業、畢業專案與維護/開發 Docusaurus 之間取得平衡。不過，這並不代表我們不想讓 Docusaurus 變得更好。事實上，**我們希望讓它變得盡可能出色**。

目前，Docusaurus 2 的實際開發工作仍在私有儲存庫中進行。不久的將來，我們會將其移至[公開儲存庫](https://github.com/facebook/docusaurus)。屆時，我鼓勵大家查看並希望以某種方式貢獻。在此之前，請保持關注 😉！

## 最後的想法

從[眾多知名專案](https://v1.docusaurus.io/en/users)採用Docusaurus作為文件系統即可看出，它對開源社群產生了重大影響。為了加速未來發展，我們正著手解決Docusaurus 1的核心問題，致力打造更完善的版本。事實上，Docusaurus 2已非紙上談兵，開發工作正如火如荼進行，可望在近期內問世。

Docusaurus的核心理念始終是讓您能快速建立開箱即用的文件網站，這個使命在Docusaurus 2中依然不變。

我們也必須聲明：**由於投入Docusaurus 2開發，我們將減少對Docusaurus 1新功能/重大變更的支援。**

若您正在使用Docusaurus，便是社群一份子，請持續提供改善建議。若認同我們的理念，歡迎透過[Open Collective](https://opencollective.com/Docusaurus)贊助Docusaurus。

> 若您透過[Open Collective](https://opencollective.com/Docusaurus)贊助，我們將親自協助您維護與升級Docusaurus網站。

最後提醒，別忘了到[GitHub](https://github.com/facebook/docusaurus)點擊**星號**與**追蹤**按鈕，並在[X](https://x.com/docusaurus)關注我們。
