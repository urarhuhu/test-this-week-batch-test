# 使用多個側邊欄

您可以為每組**需要歸類在一起**的**Markdown文件**建立專屬側邊欄。

:::tip

Docusaurus官方網站就是使用多側邊欄的絕佳範例：

- [文檔](../../../introduction.mdx)
- [API](../../../cli.mdx)

:::

請參考以下範例：

```js title="sidebars.js"
export default {
  tutorialSidebar: {
    'Category A': ['doc1', 'doc2'],
  },
  apiSidebar: ['doc3', 'doc4'],
};
```

當瀏覽`doc1`或`doc2`時，系統會顯示`tutorialSidebar`；瀏覽`doc3`或`doc4`時則會顯示`apiSidebar`。

## 理解側邊欄關聯機制 {#sidebar-association}

延續前例，若某個`commonDoc`同時出現在兩個側邊欄中：

```js title="sidebars.js"
export default {
  tutorialSidebar: {
    'Category A': ['doc1', 'doc2', 'commonDoc'],
  },
  apiSidebar: ['doc3', 'doc4', 'commonDoc'],
};
```

Docusaurus如何決定瀏覽`commonDoc`時顯示哪個側邊欄？答案是：系統無法判斷，我們也不保證會選擇哪個側邊欄。

當您將文件Y加入側邊欄X時，會建立雙向綁定：側邊欄X會顯示文件Y的連結，而瀏覽文件Y時會顯示側邊欄X。但有時我們需要打破這種隱性綁定：

1. _如何在側邊欄X建立文件Y的連結，但瀏覽Y時不顯示側邊欄X？_ 例如當文件Y同時存在於多個側邊欄（如前例），且需要明確指定顯示某個側邊欄時。
2. _如何讓瀏覽文件Y時顯示側邊欄X，但側邊欄X不包含Y的連結？_ 例如當Y是「文件首頁」而側邊欄僅用於導覽時。

Front matter選項`displayed_sidebar`可強制設定側邊欄關聯。沿用前例，您仍可使用簡化文件語法而無需特殊配置：

```js title="sidebars.js"
export default {
  tutorialSidebar: {
    'Category A': ['doc1', 'doc2'],
  },
  apiSidebar: ['doc3', 'doc4'],
};
```

然後添加front matter：

```md title="commonDoc.md"
---
displayed_sidebar: apiSidebar
---
```

這會明確指示Docusaurus在瀏覽`commonDoc`時顯示`apiSidebar`。同理，您也可以讓未包含文件Y的側邊欄X出現在文件Y頁面：

```md title="home.md"
---
displayed_sidebar: tutorialSidebar
---
```

即使`tutorialSidebar`未包含`home`的連結，瀏覽`home`時仍會顯示該側邊欄。

若設定`displayed_sidebar: null`，該頁面將不會顯示任何側邊欄，連帶也不會產生分頁導覽。

## 產生分頁導覽 {#generating-pagination}

Docusaurus使用側邊欄結構在文件頁面底部產生「下一頁」與「上一頁」分頁連結。系統嚴格依據當前顯示的側邊欄產生分頁：若無關聯側邊欄，則不會產生分頁。但需注意，「下一頁/上一頁」連結的文件可能顯示不同側邊欄：這些文件雖屬於當前側邊欄，但其front matter可能設定了不同的`displayed_sidebar`。

若透過`displayed_sidebar`front matter設定的側邊欄未包含當前文件，則不會顯示分頁導覽。

您可以使用front matter選項`pagination_next`和`pagination_prev`自訂分頁。參考以下側邊欄：

```js title="sidebars.js"
export default {
  tutorial: [
    'introduction',
    {
      installation: ['windows', 'linux', 'macos'],
    },
    'getting-started',
  ],
};
```

「windows」頁面的「下一頁」連結預設指向「linux」，這並不合理：您會希望讀者在安裝完成後進入「getting started」。此時可手動設定分頁：

```md title="windows.md"
---
# highlight-next-line
pagination_next: getting-started
---

# Installation on Windows
```

您也可透過`pagination_next: null`或`pagination_prev: null`停用特定分頁連結。

分頁標籤預設使用側邊欄標籤。您可用front matter選項`pagination_label`自訂該文件在分頁中的顯示名稱。

## `ref`類型項目 {#sidebar-item-ref}

`ref` 類型與 [`doc` 類型](./items.mdx#sidebar-item-doc)在各方面完全相同，唯一的區別在於它不會參與生成導航元數據。它僅將自身註冊為一個連結。在[生成分頁](#generating-pagination)和[顯示側邊欄](#sidebar-association)時，`ref` 項目會被完全忽略。

當您希望從多個側邊欄連結到同一份文件時，這個類型特別有用。文件僅屬於一個側邊欄（即註冊為 `type: 'doc'` 的側邊欄或來自自動生成的目錄），但其連結會出現在所有註冊過的側邊欄中。

考慮以下範例：

```js title="sidebars.js"
export default {
  tutorialSidebar: {
    'Category A': [
      'doc1',
      'doc2',
      // highlight-next-line
      {type: 'ref', id: 'commonDoc'},
      'doc5',
    ],
  },
  apiSidebar: ['doc3', 'doc4', 'commonDoc'],
};
```

您可以將 `ref` 類型視為等效於以下操作：

- 為 `commonDoc` 設置 `displayed_sidebar: tutorialSidebar`（`ref` 在側邊欄關聯中被忽略）
- 為 `doc2` 設置 `pagination_next: doc5` 並為 `doc5` 設置 `pagination_prev: doc2`（`ref` 在分頁生成過程中被忽略）
