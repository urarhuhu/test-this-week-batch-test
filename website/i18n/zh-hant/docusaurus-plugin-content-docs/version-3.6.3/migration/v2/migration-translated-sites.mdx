# 已翻譯的網站

本頁面說明如何將已翻譯的 Docusaurus v1 網站遷移至 Docusaurus v2。

## 國際化差異 {#i18n-differences}

Docusaurus v2 的國際化概念與 Docusaurus v1 大致相似，但存在一些差異。

它並未與 Crowdin 緊密耦合，您可以使用 Git 或其他 SaaS 服務替代。

### 不同的檔案系統路徑 {#different-filesystem-paths}

在 Docusaurus v2 中，本地化內容通常位於 `website/i18n/[locale]`。

Docusaurus v2 基於插件系統模組化，每個插件負責管理自己的翻譯。

每個插件都有自己的 i18n 子資料夾，例如：`website/i18n/fr/docusaurus-plugin-content-blog`

### 更新的翻譯 API {#updated-translation-apis}

在 Docusaurus v1 中，您使用 `<translate>` 來翻譯頁面：

```jsx
const translate = require('../../server/translate.js').translate;

<h2>
  <translate desc="the header description">
    This header will be translated
  </translate>
</h2>;
```

在 Docusaurus v2 中，您使用 `<Translate>` 來翻譯頁面

```jsx
import Translate from '@docusaurus/Translate';

<h2>
  <Translate id="header.translation.id" description="the header description">
    This header will be translated
  </Translate>
</h2>;
```

:::note

`write-translations` CLI 仍然可用於從代碼中提取翻譯。

代碼翻譯現在使用 Chrome i18n JSON 格式添加到 `i18n/[locale]/code.json` 中。

:::

### 更嚴格的 Markdown 解析器 {#stricter-markdown-parser}

Docusaurus v2 使用 [MDX](https://mdxjs.com/) 來解析 Markdown 檔案。

MDX 將 Markdown 檔案編譯為 React 組件，比 Docusaurus v1 的解析器更嚴格，並且會在出現錯誤時使構建失敗，而不是渲染一些不良內容。

此外，HTML 元素必須替換為 JSX 元素。

這對於國際化尤其重要，因為如果您的 Crowdin 翻譯不佳且使用了無效的標記，您的 v2 翻譯網站可能會構建失敗：您可能需要進行一些翻譯清理來修復錯誤。

## 遷移策略 {#migration-strategies}

本節將幫助您了解如何在遷移到 v2 後**保留現有的 v1 翻譯**。

對於使用 Crowdin 的 Docusaurus v1 網站，有**多種可能的策略**可以進行遷移，每種策略都有不同的權衡。

:::warning

本文檔是為了幫助您遷移而盡力提供的，如果您找到更好的方法，請幫助我們改進它！

:::

首先，我們建議：

- 在不包含翻譯的情況下將您的 v1 Docusaurus 網站遷移到 v2
- 熟悉 [Docusaurus v2 的新國際化系統](../../i18n/i18n-introduction.mdx)
- 使用一個新的、未翻譯的 Crowdin 項目和 [Crowdin 教程](../../i18n/i18n-crowdin.mdx) 讓 Crowdin 為您的 v2 網站工作

:::danger

在未理解 Crowdin 和 Docusaurus v2 國際化的情況下，請勿嘗試遷移。

:::

### 創建一個新的 Crowdin 項目 {#create-a-new-crowdin-project}

為了避免**在生產環境中破壞您的 v1 網站的風險**，一種可能的策略是複製原始的 v1 Crowdin 項目。

:::info

此策略曾用於 [升級 Jest 網站](https://jestjs.io/blog/2021/03/09/jest-website-upgrade)。

:::

不幸的是，Crowdin 沒有「複製/克隆項目」功能，這使得事情變得複雜。

- 從原始專案下載翻譯記憶庫（`.tmx`格式）（`https://crowdin.com/project/<ORIGINAL_PROJECT>/settings#tm` > `View Records`）
- 將翻譯記憶庫上傳至新專案（`https://crowdin.com/project/<NEW_PROJECT>/settings#tm` > `View Records`）
- 根據i18n文件重新配置`crowdin.yml`以適應Docusaurus v2
- 使用Crowdin CLI將Docusaurus v2源文件上傳至新專案
- 在Crowdin上將敏感字串如`id`或`slug`標記為「隱藏字串」
- 在「Translations」標籤頁，點擊「Pre-Translation > via TM」（`https://crowdin.com/project/<NEW_PROJECT>/settings#translations`）
- 先嘗試「100%匹配」（比「Perfect」會翻譯更多內容），並預翻譯你的源文件
- 下載Crowdin翻譯至本地
- 嘗試運行/建置你的網站，查看是否有錯誤

首次嘗試時很可能會遇到錯誤：預翻譯可能會嘗試翻譯不應翻譯的內容（front matter、admonition、代碼塊等），且翻譯後的MD文件可能對MDX解析器無效。

你需要修復所有錯誤直到網站能成功建置。可以通過修改本地翻譯後的MD文件來實現，並使用`docusaurus build --locale fr`一次修復一個語系的網站。

我們無法提供修復這些錯誤的終極指南，但常見錯誤通常由於：

- 未在Crowdin中將足夠的字串標記為「隱藏字串」，導致預翻譯嘗試翻譯這些字串。
- 存在不良的v1翻譯，導致v2中出現無效標記：翻譯中包含不良HTML元素和未閉合的標籤
- 任何被MDX解析器拒絕的內容，例如使用HTML元素而非JSX元素（使用[MDX playground](https://mdxjs.com/playground/)進行除錯）

你可能需要重複此預翻譯過程，最終嘗試「Perfect」選項並限制僅對某些語言/文件進行預翻譯。

:::tip

在問題Markdown元素周圍使用[`mdx-code-block`](../../i18n/i18n-crowdin.mdx#mdx-solutions)：Crowdin較不容易在代碼塊中出錯。

:::

:::note

你可能會注意到舊專案中已翻譯的內容在新專案中未翻譯。

Crowdin Markdown解析器會隨時間演進，且每個Crowdin專案的解析器版本可能不同，這可能導致預翻譯無法預翻譯所有字串。

此解析器版本未公開，你需要聯繫Crowdin支援以了解專案的解析器版本並固定特定版本。

在兩個Crowdin專案中使用相同的CLI版本和解析器版本可能會得到更好的結果。

:::

:::danger

Crowdin有「上傳翻譯」功能，但根據我們的經驗，對於Markdown效果不佳

:::

### 使用現有的Crowdin專案 {#use-the-existing-crowdin-project}

如果不介意修改現有Crowdin專案並承擔搞砸的風險，可以嘗試使用Crowdin分支系統。

:::warning

此工作流程尚未經過實際測試，請回報其效果如何。

:::

這樣就不需要創建新Crowdin專案、轉移翻譯記憶庫、應用預翻譯並嘗試修復預翻譯錯誤。

你可以為Docusaurus v2創建一個Crowdin分支，上傳v2源文件，並在準備好後將Crowdin分支合併至主分支。

### 使用Git替代Crowdin {#use-git-instead-of-crowdin}

可以遷移出Crowdin，將翻譯文件加入Git中。

使用Crowdin CLI下載v1翻譯文件，並將這些翻譯文件放置於Docusaurus v2的正確文件系統位置。
